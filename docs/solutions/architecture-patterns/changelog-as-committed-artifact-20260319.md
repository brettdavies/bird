# CHANGELOG.md as Committed Artifact

## Problem

CHANGELOG.md was auto-generated by `git-cliff` on every push to main via
the CI changelog job. This caused two problems:

1. **Wiped curated content.** Pre-conventional-commit releases (v0.1.0,
   v0.1.1) had manually written changelogs that git-cliff couldn't
   reconstruct from commit history.

2. **Empty sections from squash merges.** The release branch pattern
   (cherry-pick from development → PR to main → squash merge) collapses
   all commits into one. git-cliff only sees the squash commit title,
   producing changelogs like:

   ```markdown
   ## [0.2.0] - 2026-04-01

   ### Added

   - V0.2.0 release   ← just the PR title
   ```

## Root Cause

git-cliff regenerates the entire changelog from git commit history on
main. With squash-only merging enforced (`allow_squash_merge: true`,
all others disabled), each release PR becomes a single commit. The
individual `feat:`, `fix:`, `refactor:` commits from the release branch
are invisible on main after squash.

## Solution

CHANGELOG.md is now a **committed artifact** managed during release
preparation, not auto-generated on every push.

### Release prep flow

```text
1. Create release branch from main
2. Cherry-pick commits from development
3. Run git cliff on the release branch (individual commits visible)
4. Commit CHANGELOG.md to the release branch
5. PR to main → squash merge (CHANGELOG.md preserved in the merge)
6. Tag and push → release workflow triggers
```

### Release workflow

The release workflow extracts the latest section from the committed
CHANGELOG.md for the GitHub Release body:

```yaml
- name: Extract release notes from CHANGELOG.md
  id: changelog
  run: |
    NOTES=$(awk '/^## \[/{if(n++)exit}n' CHANGELOG.md)
    if [ -z "$NOTES" ]; then
      NOTES="Release ${GITHUB_REF_NAME}"
    fi
    {
      echo "content<<EOF"
      echo "$NOTES"
      echo "EOF"
    } >> "$GITHUB_OUTPUT"
```

### Pre-conventional releases

Releases that predate conventional commits on main (squash-merged
history) have their content preserved in `cliff.toml`'s `footer` field.
The `ignore_tags` setting prevents git-cliff from generating empty
duplicate sections for these tags:

```toml
ignore_tags = "v0\\.1\\.[01]"

footer = """
## [0.1.1] - 2026-03-17
...
## [0.1.0] - 2026-03-17
...
"""
```

### Changes made

| File | Change |
|------|--------|
| `dot-github/rust-ci.yml` | Removed changelog job entirely |
| `dot-github/rust-release.yml` | Replaced git-cliff with awk extraction |
| `bird/ci.yml` | Dropped to `contents: read`, removed secret |
| `bird/cliff.toml` | Added footer + ignore_tags for v0.1.0, v0.1.1 |

### CI workflow simplification

The CI workflow (`rust-ci.yml`) went from 4 jobs to 3:

- check (fmt, clippy, test)
- audit (cargo-deny)
- package-check (cargo publish --dry-run)

No secrets required. Only `contents: read` permission needed.

## Prevention

- When setting up a new repo, do NOT add a changelog auto-commit job
- Generate changelogs as part of the release preparation process
- Use `cliff.toml` footer for any releases with non-conventional history
- Test `git cliff` output locally before committing CHANGELOG.md
